六、计算逻辑与建表规范

以下内容合并自原 ms-warehouse-batch-data/docs/FWVD-1071 设计文档，按需求维度整理。

6.1 通用规范

ODS 命名规范

ods_{业务域}_{数据源}_{表名}_{数据增长类型}

  • 业务域：trade、sales、product、traffic、marketing、platform
  • 数据源：file、api
  • 数据增长：di（日增量）、df（日全量）

建表规范

  • 按月分区：PARTITION BY RANGE(smt)
  • 必含字段：insert_time DATETIME DEFAULT CURRENT_TIMESTAMP
  • 单机 Doris：replication_allocation = "tag.location.default: 1"
  • UNIQUE KEY 表：enable_unique_key_merge_on_write = "true"

6.2 PID 需求一：GMV 计算表

产出表：ads_sales_pid_gmv_calc_1d（见第四节 4.1）

规则

  • 输入：全部笔订单（SKU 明细）+ 全渠道商品表
  • 关联：order_item.seller_sku = product.msku → parent_asin 作为 PID
  • 筛选：Cancelation/Return Type ≠ Cancel；Order Amount ≠ 0
  • 汇总：按日期、PID、店铺

字段计算

  • 总 GMV = 营业总额 - 商家折扣 + 运费
  • 每单补贴 = 补贴 / 单量
  • 客单价 = 实付总价 / 单量（保留两位小数）

6.3 PID 需求二：数据看板

产出表：ads_sales_pid_dashboard_1d（见第四节 4.2）

规则

  • 底表：product_list（ID 即 PID）
  • 关联：全渠道商品表、产品阶段、GMV 计算表、Product campaign data、Video Performance List、tk 账号信息

挂车量

  • 商务挂车：video.product_name = product.product_title，且 creator_id 不在达人列表
  • 剪辑挂车：同上，creator_id 在达人列表

广告关联

SPLIT_PART(campaign_name, '-', -1) = product_list.ID

执行顺序：先产出 GMV 计算表，再产出数据看板。

6.4 销售需求二：订单明细与销量趋势

产出表：dwd_sales_tiktok_trend_order_detail_di（见第三节 3.1）

规则

  • 筛选：实付≠0；美区 18:00/17:00 前；欧洲不卡时间
  • 关联：订单表.asin = 商品表.asin，补 PID（parent_asin）、SPU
  • 目标：① 订单明细映射 ② 近两天差异播报 ③ AI 异常播报

6.5 销售需求一：每日订单数播报

产出表：ads_sales_order_count_store_1d（见第四节 4.3）

规则

  • 筛选：实付≠0
  • 美区：付款时间早于当日 夏令时 18:00 / 冬令时 17:00
  • 欧洲：不卡时间
  • 统计：按店铺 + 平台订单编号去重

时间截断（美区）

paid_time_local = paid_time + utc_offset_seconds
IF 店铺属于美区: cutoff_hour = 夏令时 ? 18 : 17
  保留 IF (paid_date_local = 播报日期 AND paid_hour_local < cutoff_hour) OR (paid_date_local < 播报日期)
ELSE IF 店铺属于欧洲: 保留 IF paid_date_local = 播报日期

需关联 dim_store_site_info 获取 utc_offset_hours、country_code。

企微播报示例

2月27日（美国时间）订单数：1店230 / 2店0 / 3店3 / 英区12 / 独立站0 @所有人
