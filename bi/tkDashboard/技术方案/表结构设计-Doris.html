准实时报表方案

以下方案约定：TikTok 订单（API）与积加全渠道订单（Shopify 等）均以约 10 分钟频率同步入 ODS，在此之上建设准实时 DWD → DWS/ADS，供准实时报表使用。

1. 数据源与更新频率

  • ods_trade_api_tiktok_order_di（及 order_item_di）：TikTok API 数据，调度约 10 分钟一次增量写入。
  • ods_sales_api_jijia_omnichannel_order_di（及 order_item_di）：积加全渠道订单（含 Shopify 等），调度约 10 分钟一次增量写入。

2. 准实时分层

  • ODS：沿用上表，不新增表；在表 COMMENT 中注明「支持准实时，约 10 分钟同步」。
  • DWD：准实时链路需要 DWD 层，与离线可同表（同一张表由准实时任务 10 分钟刷新），或在 COMMENT 中注明「准实时」；若需物理隔离可单独建表（见下命名约定）。
  • DWS / ADS：准实时报表层以 ADS 为主（可无 DWS），表名或 COMMENT 体现准实时（见下）。

3. 准实时表命名与注释约定

  • 表名是否带 realtime：建议 ADS 层准实时产出表在表名中带 _realtime 后缀（如 ads_sales_store_site_order_count_realtime），便于取数与运维区分；ODS、DWD 以 COMMENT 注明「准实时，约 10 分钟更新」即可，避免表名过长。
  • 与离线关系：同一业务可同时存在「离线 T+1 表」与「准实时表」：离线表名保持 _1d / _di，准实时表名用 _realtime 或 COMMENT 标明。

5. 准实时 DWD

dwd_sales_amount_detail_market_realtime（见四、4.1）：准实时与销量趋势均使用，TikTok + 积加全渠道订单写入。

dwd_sales_order_item_market_wide_realtime（见四、4.2）：全渠道订单商品大宽表，以 ASIN 为维度，约 10 分钟更新，供 GMV 计算、销量趋势。

  • dwd_sales_amount_detail_market_realtime 粒度：UNIQUE KEY(smt, sdt, store_code, store_site_code, order_id, data_source)，即每订单每店铺/站点一行；含 payment_amount、order_amount、net_amount、order_date_market / payment_date_market 等。
  • 数据来源：TikTok 订单（ODS）+ 积加全渠道订单（ODS）以约 10 分钟或小时频率写入 dwd_sales_amount_detail_market_realtime；全渠道订单明细+订单+商品写入 dwd_sales_order_item_market_wide_realtime。data_source 区分 1-积加、2-手动导入、3-第三方 API。
  • 下游：dwd_sales_amount_detail_market_realtime → dws_sales_store_market_realtime、dws_sales_store_site_market_realtime、ads_sales_store_site_order_count_realtime；dwd_sales_order_item_market_wide_realtime → dws_sales_pid_gmv_realtime、ads_sales_pid_gmv_calc_realtime。

具体 DDL 与字段以落地时为准；本方案仅约定数据源、分层、命名与复用方式。

三、ODS 层表结构

3.1 ods_trade_api_tiktok_order_di

全部笔订单主表，TikTok API 对接。用于 PID 需求一 GMV 计算、销售需求一订单数播报；支持准实时，约 30 分钟抽取一次。

表实现

本表当前实现如下：

  • 数据来源：MySQL 库 ms-cloud，表 tiktok_order（通过 Doris JDBC Catalog 连接）。
  • 抽取频率：每 30 分钟一次（海豚调度）。
  • 增量字段：update_time。
  • 调度参数：currentTime，表示本次抽取的截止时间，格式 yyyy-MM-dd HH:mm:ss（如 2026-02-03 15:15:13）。
  • 时间窗口：由 currentTime 往前推 40 分钟，即抽取 update_time 在 (currentTime − 40 分钟, currentTime] 内的记录。

  • DWS 汇总：由 DWD 按 smt、sdt、market_id（映射 store_code）、market_name（映射 store_name）、pid 聚合，产出 dws_sales_pid_gmv_realtime
  • ADS：来源 dws_sales_pid_gmv_realtime（可直接使用或轻量加工）

字段计算

  • store_code = CAST(market_id AS VARCHAR)
  • store_name = market_name
  • order_quantity = COUNT(DISTINCT order_id)
  • sale_qty = SUM(sale_qty)
  • paid_amount = SUM(按明细分摊的买家实付金额，如 item_price×sale_qty 或按比例分摊 buyer_pay_amount)
  • atv = paid_amount / order_quantity（保留两位小数）

6.5 销售需求一：每日订单数播报

取数表：ads_sales_store_site_order_count_realtime（见第六节 6.3）。播报为站点层级，该表保留历史数据，每日播报按 sdt = 播报日期 查询各站点即可，无需单独 1d 表。

规则

  • 筛选：实付≠0（在 DWD/DWS 写入或上游保证）
  • 美区：付款时间早于当日 夏令时 18:00 / 冬令时 17:00
  • 欧洲：不卡时间
  • 统计：按店铺 + 平台订单编号去重

时间截断（美区）

paid_time_local = paid_time + utc_offset_seconds
IF 店铺属于美区: cutoff_hour = 夏令时 ? 18 : 17
  保留 IF (paid_date_local = 播报日期 AND paid_hour_local < cutoff_hour) OR (paid_date_local < 播报日期)
ELSE IF 店铺属于欧洲: 保留 IF paid_date_local = 播报日期
