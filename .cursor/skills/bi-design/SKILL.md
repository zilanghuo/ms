# BI 看板文档体系

**适用范围**：本 skill 针对仓库内 **bi** 目录及其所有子目录（如 `bi/tkDashboard`、`bi/xxx`）。凡在该目录下进行原型、需求文档、接口文档、技术方案、表结构设计等编写或修改，均按本约定执行。

约定基于 `bi/tkDashboard` 提炼的文档类型、目录结构与编写惯例，在 bi 下其他子项目中复用同一套规范。

## 何时使用

- 当前工作路径在 **bi/** 下，或用户明确提到 **bi 目录**、**BI 看板**、**tkDashboard**
- 用户要求写/改**需求文档**、**接口文档**、**技术方案**、**表结构设计**、**原型页**
- 用户要求为**新接口**或**第三方对接接口**补充接口说明、请求/响应示例、字段表、**落库/导出字段备注**
- 用户要求新增**原型页面**（HTML）并挂到导航
- 用户询问文档应放在哪、章节应如何组织

## 文档类型与位置

| 类型 | 典型路径（均相对于 bi 或 bi 下子目录） | 说明 |
|------|----------------------------------------|------|
| 入口首页 | `bi/<子项目>/index.html` | 侧栏导航 + iframe 加载子页 |
| 需求文档 | `bi/<子项目>/*/需求文档.html` | 按「需求一、需求二」分章节，含优先级表、描述、流程、目标结果、链接到子页 |
| 原型页 | `bi/<子项目>/*/htmls/*.html` | 可交互的页面原型，从需求文档或首页链接进入 |
| 接口文档 | `bi/<子项目>/接口设计文档/接口文档.html` | 通用约定 + 按模块的接口列表，含请求/响应、字段表、示例 |
| 技术方案 | `bi/<子项目>/技术方案/技术方案.html` | 方案说明、数据流、表设计概要 |
| 表结构设计 | `bi/<子项目>/技术方案/表结构设计-*.html` | 建表 DDL、字段说明（可带代码块样式） |

**约定**：以 **HTML 单文件** 为主，保留 `.html` 作为唯一正式版本；若存在同名的 `.md`，可移除仅保留 `.html`。

## 统一样式（HTML 文档）

所有「文档类」HTML（需求文档、接口文档、技术方案、表结构设计）共用一套样式约定，便于风格一致：

- 根变量：`--bg`、`--card`、`--text`、`--muted`、`--accent`、`--border`
- 主容器：`.doc`，max-width 1280px，圆角 + 边框
- 标题：`h1` 底部 accent 线，`h2`/`h3` 层级清晰
- 表格：外包一层 `.table-wrap`（overflow-x: auto），`table` 边框、表头背景 `#f1f5f9`
- 代码：`code` 浅底圆角；`pre` 包 `code` 用于多行请求/响应，背景 `#f8fafc`
- 返回链接：`.back a` 使用 `--accent` 色

新增文档时直接复用现有任意一份文档的 `<style>` 块即可。

## 接口文档编写惯例

1. **顶部**：标题 + 简短说明（对应哪些页面/模块）。
2. **通用约定**：Base URL、请求/响应格式、分页、时间格式、鉴权、统一响应结构（如 `code`/`message`/`data`）。
3. **按模块分 h2**：如「零、下拉选项」「一、需求一：店铺统计」「五、积加对接：全渠道商品」。
4. **每个接口包含**：
   - 接口地址：`POST/GET URL`（若为第三方，写完整 URL）
   - 请求：请求体示例（JSON）+ 参数表（参数、类型、说明）
   - 响应结构：根字段说明 + `data` 下字段说明
   - 列表类：若有 `rows[]` 或 `childList[]`，单独列出「单条对象字段」表（字段名、类型、说明）
   - 响应 data 示例：至少一条完整或代表性示例（含嵌套对象时填满关键字段）
   - **落库/导出字段备注**（若需要）：明确「从哪条路径取哪些字段」用于导出 Excel 或落库。
5. **接口总览**：文末用表格列出序号、页面、功能、Method、路径，序号从 1 连续编号。

## 第三方对接接口（如积加）

- 单独一个 h2（如「五、积加对接：全渠道商品接口」）。
- 请求体示例 + 参数表。
- 响应：根结构（traceId、code、success、data 等）+ `data` 内 `total`、`rows`、`condition` 等。
- `data.rows[]` 或 `data.rows[].childList[]` / `itemVoList[]`：各做一张「单条对象字段」表，再在响应示例中填一条完整示例。
- 最后一段：**落库所需数据及字段**——分「1. 订单数据：字段列表」「2. 订单商品明细：字段列表」，并注明来源路径（如 `data.rows[]`、`data.rows[].itemVoList[]`）。

## 需求文档编写惯例

- 每个需求一个 h2（需求一、需求二…）。
- 每个需求下可设「优先级/影响面/价值」小表（`.req-metrics` 或普通 table）。
- 子节：需求描述（数据范围、筛选规则、统计口径）、流程（有序列表）、最终目标结果（示例表或说明）。
- 需跳转原型时：在需求文档中写链接到 `htmls/xxx.html`，原型页内「返回」链回需求文档。

## 表结构 / 技术方案

- 表结构设计：表名、说明、建表 DDL（`<pre><code>`），字段列表可用表格或 DDL 内注释。
- 技术方案：数据流、分层（ODS/DWD/DWS/ADS）、与需求/接口的对应关系，可配图或表格。
